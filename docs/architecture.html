Architecture · Baguette

Thin CLI, rich domain, narrow ports

A headless iOS Simulator console. Five layers sit on CoreSimulator + SimulatorKit. The same Domain + Infrastructure powers three consumers: the CLI (baguette stream / input), the asc-pro plugin's spawned subprocess pipeline, and the standalone baguette serve web UI.

THREE CONSUMERS

One Domain, three frontends

  • CLI — baguette stream writes encoded frames to stdout; baguette input reads JSON gestures from stdin. Both run as standalone processes.
  • asc-pro plugin — spawns one persistent baguette input subprocess per booted device; HTTP routes translate browser events to JSON lines on the subprocess's stdin.
  • baguette serve — same Server / GestureDispatcher / Stream pipeline in-process; one bidirectional WebSocket per stream carries binary frames down + text JSON (gestures + control) up.

FOCAL · L4

Rich domain, narrow ports

Aggregates as protocol + Live impl: Simulators / LiveSimulators, Chromes / LiveChromes. Simulator values carry rich verbs (boot, screen, input, chrome(in:)) that delegate via the injected aggregate. Gesture (Tap, Swipe, Touch1, Touch2, Press, Scroll, Pinch, Pan) — each is one file with wireType + parse + execute(on: Input). Adding a new gesture is one file + one register line.

PRIVATE-API BLAST RADIUS

ObjC runtime, no ObjC target

CoreSimulators, IndigoHIDInput, and SimulatorKitScreen are the only files that reach into CoreSimulator / SimulatorKit. Each uses dlopen + class_getMethodImplementation + unsafeBitCast from pure Swift — no separate ObjC target, no bridging header. The 9-arg IndigoHIDMessageForMouseNSEvent recipe is heavily commented inline.
